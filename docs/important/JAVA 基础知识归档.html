# java基础知识

1、栈、堆、队列的区别 1、栈是先进后出，在表尾一端插入和删除
2、队列是先进先出，在表头删除，表尾插入
3、栈存放基本类型的变量数据和对象的引用，堆存放new出来的对象
2、接口和抽象类的区别 1、抽象类用abstract修饰,接口用interface
2、抽象类可以包含普通方法和字段，子类继承时不用重写
3、一个子类只能继承一个抽象类，可以实现多个接口
4. JDK1.8 接口中可以用default修饰静态方法

# 集合框架

1、常见的集合有哪些 1、Collection接口的子接口包括：Set接口和List接口
2、Map接口的实现类主要有：HashMap、TreeMap、Hashtable、ConcurrentHashMap以及Properties等
3、Set接口的实现类主要有：HashSet、TreeSet、LinkedHashSet等
4、List接口的实现类主要有：ArrayList、LinkedList、Stack以及Vector等
2、HashMap与HashTable的区别？ 1、HashMap没有考虑同步，是线程不安全的；Hashtable使用了synchronized关键字，是线程安全的
2、HashMap允许K/V都为null；后者K/V都不允许为null
3、HashMap继承自AbstractMap类，而Hashtable继承自Dictionary类
3、ArrayList和LinkedList的区别？ 1、LinkedList 实现了 List 和 Deque 接口，一般称为双向链表；ArrayList 实现了 List 接口，动态数组
2、LinkedList 在插入和删除数据时效率更高，ArrayList 在查找某个 index 的数据时效率更高
3、LinkedList 比 ArrayList 需要更多的内存
4、HashSet是如何保证数据不可重复的？ 1、HashSet的底层其实就是HashMap，只不过我们HashSet是实现了Set接口并且把数据作为K值，而V值一直使用一个相同的虚值来保存
2、由于HashMap的K值本身就不允许重复，并且在HashMap中如果K/V相同时，会用新的V覆盖掉旧的V，然后返回旧的V，那么在HashSet中执行这一句话始终会返回一个false，导致插入失败，这样就保证了数据的不可重复性
5、HashMap在JDK1.7和JDK1.8中有哪些不同？ 1、在jdk1.8中新增了数组转红黑树,详见
传送门
6、ConcurrentHashMap在JDK1.7和JDK1.8中有哪些不同？ 1、数据结构发生变化
2、优化了实现线程安全的方式,详见
传送门
7、HashMap的长度为什么要是2的幂次方 1. 提高哈希计算的效率
2. 减少哈希冲突
传送门
8、为什么HashMap中String、Integer这样的包装类适合作为K？Object作为K应该怎么办？ 1、String、Integer等包装类都是final类型，保证key的不可更改性
2、内部已重写了equals()、hashCode()等方法，不容易出现Hash值计算错误的情况
1、重写hashCode()，equals()方法，hashCode()
2、hashCode()必须是完整的对象信息，减少hash碰撞
3、equals()，x.equals(null)必须返回false

# 缓存

1、为什么要使用Redis，Redis有几种数据类型 1、内存数据库，速度快性能好，单线程操作，采用了非阻塞I/O多路复用机制，并发高
1、String 整数，浮点数或者字符串
2、Set 集合
3、Zset 有序集合
4、Hash 散列表
5、List 列表
2、Redis持久化机制，Redis的过期策略 1、RDB 持久化-将某个时间点的数据快照放在硬盘上
2、AOF 持久化-定时或每次写入数据时，将更新内容同步到磁盘，提供多种模式
1、volatile-lru 从已设置过期时间的数据集中挑选最近最少使用的数据淘汰
2、volatile-ttl 从已设置过期时间的数据集中挑选将要过期的数据淘汰
3、volatile-random从已设置过期时间的数据集中任意选择数据淘汰
4、allkeys-lru从所有数据集中挑选最近最少使用的数据淘汰
5、allkeys-random从所有数据集中任意选择数据进行淘汰
6、noeviction禁止驱逐数据

4、什么是缓存穿透，如何避免，什么是缓存雪崩，如何避免 1、缓存穿透-当客户端发起redis和底库都不存在的数据请求
2、解决方案：查询条件设定范围，底库请求不到时在redis中设置key-null，防止重复请求
1、缓存雪崩-大量数据在同一时间失效
2、设置过期时间随机
5、如何保证缓存与数据库的读写一致性 1、最终一致性方案：
1.监听数据库的binlog到消息队列，更新缓存。
2.Cache Aside Pattern，读的时候先读库，没有就读数据库，取出数据放入缓存，响应 更新的时候先更新数据库，然后删除缓存，为防止删除失败，做延时双删或消息队列
2、同步一致性方案
1.缓存加锁，读写请求做队列
